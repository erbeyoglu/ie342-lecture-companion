Could we just enumerate everything? Slides 3–4

A 0/1 problem with n variables has 2ⁿ solutions. Drag the slider and watch the count — the y-axis below is logarithmic, so a straight line means explosive growth. The slides anchor two cases: n = 20 (still doable by computer) and n = 100 (hopeless).

2ⁿ = — ≈ —

At n = 20 there are 2²⁰ = 1,048,576 candidate solutions — a computer can grind through them. At n = 100 there are about 1.268 × 10³⁰ — beyond any machine, and it is easy to build combinatorial problems with more solutions than atoms in the universe. That is why the rest of this week replaces brute force with branch and bound: implicit enumeration that discards huge sets of candidates using bounds.

Fathom or branch? You are the algorithm Slides 23–24

A node is fathomed (never branched again) in exactly three situations. Each case below is a real node from Example 1 — pick a case, then pick the correct action. The counter tracks first-try hits.

first-try score: 0 / 4

Read the case, then commit to an action.

For a max problem the LP value of a node is an upper bound on everything below it — it can only get worse as constraints pile up. That single fact powers all three fathoming rules.
spoiler — open after the class has answered A → fathom, infeasible. B → fathom, integer and 100 > 99 so it becomes the new incumbent. C → branch: 101.8 > 100 and fractional, the branch may still hide something better than 100. D → fathom by bound: 90.33 < 100, nothing below can recover.
