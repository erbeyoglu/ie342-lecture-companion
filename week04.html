One binary, two bakery rules — either–or and if–then Slides 3–8

The bakery (Examples 12–13): 8 doughnut varieties, at most 30 dozen in total. Rule 1: varieties 1–3 are worth baking only if together they reach at least 10 dozen — so their sum s = x₁+x₂+x₃ must be 0 or at least 10. Rule 2 (added later): if variety 5 exceeds 3 dozen, variety 7 must reach 5 dozen. Play with each rule and watch a single 0–1 variable do the logic.

M = 30 s − M·y = 0 ✓ (need ≤ 0) s − 10y = 0 ✓ (need ≥ 0)
score: 0 / 0 streak: 0

Drag the s marker. Challenge: find a feasible plan with 0 < s < 10 — any y allowed.

Example 12 links s to y via s − 30y ≤ 0 and s − 10y ≥ 0 (slide 5); Example 13 adds x₅ − 3 − 30u ≤ 0 and x₇ − 5u ≥ 0 with a second binary u (slide 8). Profits pᵢ stay symbolic on the slides, so both games are about feasibility logic, not the objective. In the if–then game the other six varieties are held at 0, so the 30-dozen oven cap is not binding on the 8×8 view.
spoiler — open after the class has answered Either–or: y = 0 collapses s to exactly 0, y = 1 opens 10 ≤ s ≤ 30 — together {0} ∪ [10, 30], and 1–9 dozen is unreachable under every y. M = 30 is the smallest safe constant because the oven cap already forces s ≤ 30; a smaller M (try 20) wrongly cuts real plans, a bigger one stays valid here but big-M's should always be as tight as possible. If–then: a point (x₅, x₇) is bakeable iff x₅ ≤ 3 or x₇ ≥ 5 — with u = 0 the first constraint forces x₅ ≤ 3, with u = 1 the second forces x₇ ≥ 5. That is exactly the contrapositive of "x₅ > 3 ⇒ x₇ ≥ 5". Note the boundary: x₅ = 3 needs no promise on x₇ ("more than 3" is strict).

Can you cheat Euing's purchase-cost curve? Slides 9–12

Example 14: Euing can buy up to 1500 extra gallons of oil 1 — the first 500 at 25¢/gal, the next 500 at 20¢, the last 500 at 15¢. The slides replace the piecewise cost c(t) by z₁c(0) + z₂c(500) + z₃c(1000) + z₄c(1500) with t = Σzᵢbᵢ, Σzᵢ = 1. Pick which two z's may be positive, then slide t — when is the price the model pays the real price?

z = — model pays — true cost — biggest discount found: $0.00

The breakpoint costs come straight from the stated block prices: c(0) = $0, c(500) = $125, c(1000) = $225, c(1500) = $300. Profit = revenue − cost, so every dollar the model shaves off the cost curve is a fake dollar of profit the optimizer will chase.
spoiler — open after the class has answered Only adjacent pairs stay on the curve — that is what slide 12's constraints z₁ ≤ y₁, z₂ ≤ y₁+y₂, z₃ ≤ y₂+y₃, z₄ ≤ y₃ with y₁+y₂+y₃ = 1 enforce. Because the marginal price falls (25 → 20 → 15¢), c(t) is concave and every non-adjacent chord runs below it: the z₁ & z₄ chord gives the biggest cheat, a flat $25 discount for any t between 500 and 1000. Without the y's, the LP would always take that deal.

The subtour trap — are in/out constraints enough? Slides 15–16

Slide 15 asks: do "leave every city once, enter every city once" suffice? Compare a genuine tour with the cheapest solution those constraints actually allow. Then try to certify each with the slide-16 constraint uᵢ − uⱼ + 5xᵢⱼ ≤ 4 (i, j = 2,…,5, i ≠ j): choose u-values and press Check. Cities are numbered as in the table: 1 = GA, 2 = FW, 3 = EV, 4 = TH, 5 = SB.

Both pictures satisfy every slide-15 constraint — yet only one is a real tour. The two-loop “solution” is 294 miles cheaper, so the naive IP would return it!

x₁₂ = x₂₅ = x₅₁ = 1 and x₃₄ = x₄₃ = 1 gives GA→FW→SB→GA (269 mi) plus EV⇄TH (226 mi): 495 miles in total, and it is the true optimum of the assignment-constraints-only model. u₁ never appears — the constraints only chain cities 2,…,5.
spoiler — open after the class has answered For the one-loop solution u = (1, 2, 3, 4) works: every used arc among cities 2–5 asks unext ≥ uprev + 1, and the loop breaks that chain by passing through GA. The two-loop solution can never be certified: EV⇄TH forces both u₃ − u₄ + 5 ≤ 4 and u₄ − u₃ + 5 ≤ 4; adding them gives 10 ≤ 8. Any loop that avoids city 1 dies the same way — summing the constraint around its k arcs gives 5k ≤ 4k. That is exactly why slide 16's subtour-elimination constraints make the formulation valid.

spoiler — open after the class has answered The reformulation on slide 18 makes the domain valid: x₁ = 0y₁ + 1y₂ + 4y₃ + 6y₄ with y₁ + y₂ + y₃ + y₄ = 1, yᵢ ∈ {0,1} — the same select-one-breakpoint trick as the piecewise-linear z's, with the y's now picking a point instead of a segment. Best point per column: z = 4 at (0, 2), z = 7 at (1, 3), z = 12 at (4, 4), z = 10 at (6, 2) — the optimum is (x₁, x₂) = (4, 4) with z = 12. Note x₁ = 6 leaves only the single point (6, 2), squeezed between x₁ − x₂ ≤ 4 and x₁ + x₂ ≤ 8.

Can an IP solve Sudoku and Futoshiki? Slides 19–20

The Week-1 game returns: now that you can model with binaries, could an IP solve these boards? Take xr,c,k = 1 if the cell in row r, column c holds digit k. Pick the constraint that models each rule — wrong picks get a hint. (The patterns are generic; the slides pose the question as an open discussion.)

rules modeled: 0 / 4

Rule 1. Every cell holds exactly one digit.

Rule 2. Digit k appears exactly once in every column.

Rule 3. Futoshiki clue: the cell (r, c) must hold a GREATER digit than its right neighbour (r, c+1).

Rule 4. Sudoku asks for any valid completion. What should the objective be?

Pre-filled givens are handled by fixing their variable to 1 (e.g. a printed 8 in the corner: x1,1,8 = 1). Futoshiki is the same skeleton on a 5×5 board with digits 1–5, no box constraints, plus one inequality per clue.
spoiler — open after the class has answered Yes — both puzzles are small IPs. Sudoku: 729 binaries and four "exactly one" families (cell, row, column, 3×3 box) plus fixed givens; any feasible point is a solved board, so the objective is irrelevant. Futoshiki adds clue constraints on cell values Σk k·x. The lesson of the week: binaries model logic — either–or, if–then, restricted domains, and even newspaper puzzles.
